GPU Direct Access to Emulated NVMe via IOVA Mappings - BREAKTHROUGH SUCCESS

This commit represents a major milestone: successfully demonstrating GPU
direct access to an emulated NVMe device running in QEMU through IOVA
(I/O Virtual Address) mappings.

ACHIEVEMENT:
- GPU successfully read/wrote emulated NVMe doorbell registers
- Zero IOMMU page faults
- GPU compute working in VM with correct QEMU configuration
- Complete QEMU P2P infrastructure implemented

QEMU CHANGES (Applied to /home/stebates/Projects/qemu):

1. NEW: hw/nvme/nvme-p2p.{h,c}
   - P2P infrastructure for emulated NVMe devices
   - Direct VFIO ioctl usage for IOMMU mappings

KEY TECHNICAL BREAKTHROUGHS:

1. HSA Memory Mapping for IOVA Access
   - IOVAs cannot be directly dereferenced by GPU
   - Must use hsa_amd_memory_lock_to_pool() to create GPU mapping
   - This was THE critical missing piece

2. Lazy Initialization Pattern
   - Prevents GPU bricking during VM boot
   - P2P setup deferred until guest issues vendor command
   - Clean separation of concerns

3. Direct VFIO ioctl Usage
   - vfio_container_dma_map() is for guest RAM, not QEMU memory
   - Must use ioctl(VFIO_IOMMU_MAP_DMA) directly
   - Requires container_of to access VFIOContainer from VFIOContainerBase

4. Correct QEMU Configuration for GPU Compute
   - WORKS: -cpu EPYC (no explicit IOMMU device)
   - FAILS: -cpu host + intel-iommu,caching-mode=on
   - Critical for GPU kernel execution in VM

ARCHITECTURE:
- Host IOMMU creates IOVA mappings for GPU (via VFIO)
- QEMU allocates host memory buffers (HVA)
- QEMU maps HVAs to IOVAs via VFIO_IOMMU_MAP_DMA
- Guest issues vendor command to get IOVA addresses
- Guest uses HSA to make IOVAs GPU-accessible
- GPU accesses memory → IOMMU translates → QEMU buffers
